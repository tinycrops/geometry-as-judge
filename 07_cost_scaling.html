Experiment 6: Cost Analysis

Every metric has a cost. TokenF1 is free but blind to semantics. LLM-as-judge captures nuance but burns tokens. EmbeddingMetric occupies a previously empty niche: semantic signal at near-zero cost. This section quantifies the economics.

Cost Model

  • EmbeddingMetric: 2 Gemini Embedding 2 calls per metric evaluation (gold + predicted). Priced at $0.00001 per 1,000 characters. A typical QA answer is ~50 characters, so each pair costs roughly $0.000001.
  • LLM-as-judge: One LLM call per evaluation with ~200 token prompt + ~50 token response. Input and output token pricing varies by model.
  • TokenF1: Local string comparison — zero API cost, zero latency overhead.

Metric Call Counts by Optimizer

The total number of metric evaluations depends on the optimizer configuration. Here is the calculation logic for each optimizer we tested:

configs = {
    "Bootstrap (our config)": {
        # Each candidate: bootstrap (train * bootstrap_rounds) + eval (test)
        # Total: candidates * (train * bootstrap_rounds + test)
        "metric_calls": 3 * (30 * 2 + 50),  # = 330
    },
    "MIPROv2 light": {
        # Bootstrap phase + 6 trials * eval
        "metric_calls": 30 * 3 + 6 * 50 + 6 * 30,  # = 570
    },
    "MIPROv2 medium": {
        "metric_calls": 30 * 3 + 12 * 50 + 12 * 30,  # = 1050
    },
}

Bootstrap generates candidates separate programs, each bootstrapped over train * bootstrap_rounds examples, then evaluated on the test set. MIPROv2 adds an initial bootstrap phase followed by multiple Optuna trials, each of which evaluates on both train and test splits.

Insight

For a MIPROv2 medium run, the choice is: $0.0013 (embedding) vs $1.05 (GPT-4o judge). That is an 833x cost difference. For creative tasks where F1 gives zero signal, embedding is the ONLY option at ANY cost.

Experiment 7: MIPROv2 Optimizer Scaling

Motivation

The Bootstrap experiment (Experiment 3) showed EmbeddingMetric and TokenF1 tied at 42% EM on HotPotQA. Both metrics selected identical demonstrations and produced identical programs. This raised the question: does the metric choice ever actually matter for optimization?

The hypothesis: MIPROv2 optimizes both instructions and demonstrations. The instruction proposals are guided by the metric signal — the optimizer proposes candidate instructions, evaluates them with the metric, and uses Bayesian optimization to search the instruction space. EmbeddingMetric's smoother gradients should produce better instruction proposals than TokenF1's noisy binary signal.

Setup

  • MIPROv2 with auto="light" (10 trials) on HotPotQA multi-hop QA
  • 30 training examples, 50 test examples
  • Compared TokenF1 vs EmbeddingMetric as the optimization signal
  • Model: gpt-5.4-mini
  • Evaluation: exact match (EM), token-level F1, embedding similarity — all computed on the same held-out test set

The Mechanism: Instruction Optimization

Discovery

EmbeddingMetric generated the instruction: "You are an expert trivia master responsible for answering questions by providing a logical reasoning process." TokenF1's noisy signal caused the optimizer to fall back to the default instruction.

The mechanism behind this result is grounded in how MIPROv2 searches the instruction space:

  1. Bayesian optimization (Optuna). MIPROv2 uses a Tree-structured Parzen Estimator (TPE) to model the relationship between instruction candidates and metric scores. Each trial proposes a new instruction, evaluates it against the metric on the training set, and feeds the result back to the surrogate model.
  2. TokenF1 is noisy. Small wording changes cause binary 0/1 score jumps. The answer "Paris" scores 1.0 against gold "Paris", but "The capital of France is Paris" scores only 0.25 due to extra tokens diluting F1. These discontinuities make Optuna's surrogate model unreliable — the GP cannot distinguish between a genuinely better instruction and a lucky wording match.
  3. EmbeddingMetric is smooth. Similar answers get similar scores. "Paris" scores 0.73, "The capital city of France is Paris" scores 0.82, "Berlin" scores 0.71. The surrogate model receives clean signal and converges faster to better instruction candidates.
  4. Result: With embedding metric, the TPE model identifies that instructions encouraging structured reasoning produce consistently higher scores. With TokenF1, the signal-to-noise ratio is too low for 10 trials to distinguish good instructions from bad ones, so the optimizer defaults to the baseline instruction.

Note

MIPROv2 scores lower than Bootstrap here because auto="light" uses only 10 trials — a deliberately constrained search. The point is not absolute performance but the +4% EM differential between metrics, which was invisible with Bootstrap.

Bootstrap selects demonstrations by filtering: does this example pass the metric threshold? Both TokenF1 and EmbeddingMetric agree on pass/fail for most HotPotQA examples, producing identical demo sets and identical programs. MIPROv2 uses the metric as a continuous optimization signal, exposing the quality difference between metrics.

The Smoothness Argument

The core claim reduces to a well-known principle from continuous optimization theory: smooth objective landscapes are strictly easier to optimize than noisy ones.

Consider three candidate answers evaluated against the gold answer "Paris":

Candidate answerTokenF1EmbeddingMetric
"Paris"1.000.73
"The capital city of France is Paris"0.250.82
"Berlin"0.000.71

TokenF1 ranks the exact string match highest and penalizes the more informative answer severely. EmbeddingMetric correctly ranks the elaborated correct answer above both the terse match and the wrong city. For any optimizer that samples the metric landscape — whether gradient-based, Bayesian, or evolutionary — the embedding metric provides a more faithful and navigable surface.

This is not a theoretical concern. The MIPROv2 experiment demonstrates the practical consequence: 10 Optuna trials are sufficient for the TPE to find a good instruction with smooth signal, but insufficient with noisy signal.

Open Questions

  • Trial count scaling. Does the MIPROv2 advantage scale with trial count? Would auto="medium" (25 trials) amplify the gap, or does TokenF1 eventually catch up given enough samples?
  • Hybrid metrics. Can we combine metrics — e.g., 0.5 * F1 + 0.5 * embedding — to get factual precision from F1 AND smooth optimization from embedding?
  • SIMBA interaction. DSPy's SIMBA optimizer uses mini-batch stochastic ascent. How does metric smoothness interact with stochastic gradient estimation? Smooth metrics should reduce variance in gradient estimates.
  • Task-specific breakpoints. At what task complexity does the metric choice start to matter for Bootstrap? Creative tasks (story-01) already showed this; what about structured extraction or code generation?
