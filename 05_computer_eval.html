computer-01

Deep technical dive — semantic target evaluation using triangular anchors in interleaved embedding space. Part of ascii-bench.

The Evaluation

computer-01 is the first ascii-bench evaluation with a semantic target. Where noise-01 measured whether generated grids land in a diffuse noise region, computer-01 asks a harder question: can the model generate ASCII art that lands near a specific corpus centroid in Gemini Embedding 2's unified space?

The task: given a computer hardware subject (keyboard, monitor, floppy disk, etc.), generate ASCII art that is recognizably that hardware. The evaluation embeds the generated art using interleaved text+image embedding and measures its geometric position relative to three anchor points.

18 hardware subjects are split into 12 training and 6 held-out test subjects:

SplitSubjects
Train (12)keyboard, monitor, floppy disk, computer tower, mouse, laptop, printer, USB drive, headphones, webcam, ethernet cable, hard drive
Test (6)microchip, server rack, joystick, CD-ROM, router, speaker

The train/test split is intentional: test subjects are hardware the reference corpus does not contain. The metric must generalize — a good server rack drawing should land near the computer centroid even though the corpus has no server rack example.

The Evaluation Triangle

computer-01 positions every generated piece of ASCII art relative to three anchor points in embedding space:

  1. Noise centroid — the mean embedding of 5 random keyboard-symbol grids (shared with noise-01). Represents meaningless character soup.
  2. Blank centroid — the mean embedding of 8 minimal structural frames (empty boxes, bezels). Represents structure without semantic content.
  3. Computer centroid — the mean embedding of 6 hand-crafted computer hardware ASCII art pieces. Represents the target: dense, recognizable hardware art.

The geometric meaning is straightforward: a good ASCII art keyboard should be closer to the computer centroid than to the noise centroid. The metric normalizes against the noise baseline so that a piece scoring 0.0 is indistinguishable from noise, and 1.0 sits exactly at the computer centroid.

geometric_score = (sim_to_computer - baseline) / (1 - baseline)

where baseline = cosine(noise_centroid, computer_centroid). This normalization gives absolute positioning in embedding space, not just relative similarity.

Note
The triangle is not just a scoring trick. It defines a coordinate system in embedding space. Any new ASCII art — whether it depicts a router, a cat, or pure gibberish — gets a position relative to all three anchors simultaneously.

The corpus was designed with three properties in mind: recognizability (each piece is identifiable as its hardware subject), structural density (heavy use of box-drawing characters), and diversity (different shapes, aspect ratios, and character choices).

Internal cohesion is measured as the mean pairwise cosine similarity among all corpus embeddings. A tight cluster means the centroid is a stable target. If cohesion is low, the centroid is an average of scattered points and less meaningful as a target.

The Metric: ComputerArtMetric

The combined metric blends structural quality checks with geometric positioning in embedding space:

  • 40% structural score — non-empty, multi-line, uses structural characters, minimum width
  • 60% geometric score — cosine similarity to the computer centroid, normalized against the noise baseline

In optimization trace mode (when used inside a DSPy optimizer), the metric returns a boolean: score >= 0.35. This threshold gates which demonstrations the optimizer keeps.

Structural score

def structural_score(text: str) -> float:
    """Basic quality check: non-empty, multi-line, uses structural chars."""
    if not text or not text.strip():
        return 0.0

    lines = text.strip().splitlines()

    # Multi-line bonus
    line_score = min(1.0, len(lines) / 3.0)

    # Structural character density (box-drawing, pipes, dashes, etc.)
    structural_chars = set("|-+/\\._[]{}()=<>*#@~`'^")
    total = sum(len(l) for l in lines)
    if total == 0:
        return 0.0
    struct_count = sum(1 for c in text if c in structural_chars)
    density = struct_count / total

    # Minimum width check
    avg_width = sum(len(l) for l in lines) / len(lines)
    width_score = min(1.0, avg_width / 8.0)

    return (line_score * 0.4 + density * 0.3 + width_score * 0.3)

Three sub-components, equally intuitive: does it have multiple lines (line_score), does it use structural ASCII characters (density), and is it wide enough to depict something (width_score). The weights 0.4/0.3/0.3 were chosen to prioritize multi-line output — a single-line response is almost certainly not valid ASCII art.

Full metric class

class ComputerArtMetric:
    """
    Score ASCII art by proximity to computer centroid (away from noise).

    geometric_score = (sim_to_computer - baseline) / (1 - baseline)
    where baseline is the noise centroid's sim to computer centroid.

    Combined: 0.4 * structural + 0.6 * geometric
    """

    def __init__(self, computer_centroid, noise_centroid, threshold=0.35):
        self.computer_centroid = computer_centroid
        self.noise_centroid = noise_centroid
        self.baseline = cosine(noise_centroid, computer_centroid)
        self.threshold = threshold
        print(f"  Metric baseline (noise-computer): {self.baseline:.4f}")

    def __call__(self, example, pred, trace=None):
        art = str(getattr(pred, "ascii_art", ""))

        if not art.strip():
            score = 0.0
        else:
            struct = structural_score(art)
            if struct < 0.1:
                score = struct * 0.4
            else:
                try:
                    png = render(art)
                    vec = embed_interleaved(art, png)
                    sim_computer = cosine(vec, self.computer_centroid)
                    sim_noise = cosine(vec, self.noise_centroid)

                    # Geometric: how much closer to computer than to noise
                    geo = np.clip(
                        (sim_computer - self.baseline) / (1.0 - self.baseline),
                        0.0, 1.0
                    )
                    score = 0.4 * struct + 0.6 * float(geo)
                except Exception as e:
                    print(f"  Embed error: {e}")
                    score = struct * 0.4

        if trace is not None:
            return score >= self.threshold
        return score

    @property
    def __name__(self):
        return "ComputerArtMetric"

Key implementation details:

  • If structural score falls below 0.1, the metric short-circuits and returns only the structural component — no point embedding garbage.
  • The trace is not None branch converts the continuous score to a boolean for DSPy optimizers. The 0.35 threshold means only demonstrations scoring above 35% are retained during bootstrap.
  • Embedding errors (rate limits, API failures) fall back to structural-only scoring rather than crashing the evaluation.

Interleaved Embedding

The core infrastructure that makes computer-01 work is embed_interleaved — a function that embeds text and its rendered PNG together in a single call to Gemini Embedding 2:

def embed_interleaved(text: str, png: bytes, max_retries: int = 5) -> np.ndarray:
    """Embed text + rendered image together in Gemini's unified space."""
    for attempt in range(max_retries):
        try:
            r = client.models.embed_content(
                model=EMBED_MODEL,
                contents=[types.Content(parts=[
                    types.Part(text=text),
                    types.Part.from_bytes(data=png, mime_type="image/png"),
                ])]
            )
            return np.array(r.embeddings[0].values, dtype=np.float32)
        except Exception as e:
            if "429" in str(e) or "RESOURCE_EXHAUSTED" in str(e):
                wait = min(60 * (attempt + 1), 120)
                print(f"  Rate limited, waiting {wait}s ...")
                time.sleep(wait)
            else:
                raise
    raise RuntimeError("Max retries exceeded for embedding")

The interleaved embedding captures both the semantic text content and the visual appearance of the rendered art. This is critical: a piece of text that says "keyboard" but renders as random dots would score differently than one that both says "keyboard" and looks like a keyboard when rendered to pixels.

The rendering pipeline (render()) converts ASCII text to a PNG using DejaVu Sans Mono at 48pt with 24px padding. This standardizes the visual input: every piece of art is rendered the same way before embedding.

def render(text: str) -> bytes:
    """Render ASCII art to PNG."""
    lines = text.splitlines()
    if not lines:
        lines = [" "]
    test = Image.new("RGB", (1, 1))
    d = ImageDraw.Draw(test)
    lh = FONT.getbbox("A")[3] + 6
    mw = max((d.textlength(l, font=FONT) for l in lines), default=10)
    img = Image.new("RGB",
        (int(mw) + PAD * 2, lh * len(lines) + PAD * 2),
        (255, 255, 255))
    draw = ImageDraw.Draw(img)
    y = PAD
    for line in lines:
        draw.text((PAD, y), line, fill=(0, 0, 0), font=FONT)
        y += lh
    buf = io.BytesIO()
    img.save(buf, format="PNG")
    return buf.getvalue()

Retry logic handles Gemini's rate limiting (429/RESOURCE_EXHAUSTED) with exponential backoff up to 120 seconds. This is necessary because each evaluation call triggers an embedding request, and a full evaluation run with 6 test subjects plus bootstrap optimization can consume hundreds of API calls.

DSPy Program: ComputerArtGenerator

The generation program is a straightforward ChainOfThought module with a signature that maps (subject, description) to ascii_art:

class ComputerArtGenerator(dspy.Module):
    """Generate ASCII art of computer hardware."""

    def __init__(self):
        self.generate = dspy.ChainOfThought(
            dspy.Signature(
                "subject, description -> ascii_art",
                instructions=(
                    "Generate ASCII art of the specified computer hardware. "
                    "Use box-drawing characters (|, -, +, /, \\, _, .), brackets, "
                    "and other structural ASCII characters to create a recognizable "
                    "representation. Keep it compact (5-10 lines). "
                    "The art should clearly look like the hardware described."
                )
            )
        )

    def forward(self, subject, description):
        return self.generate(subject=subject, description=description)

The instructions explicitly guide the model toward box-drawing characters. This is not just stylistic preference — the structural score component rewards density of these characters, and the computer reference corpus is built from them. The instructions align the generator with what the metric measures.

The dataset construction splits 18 hardware subjects into 12 train / 6 test:

def make_dataset():
    trainset = []
    testset = []
    for i, item in enumerate(HARDWARE_SUBJECTS):
        ex = dspy.Example(
            subject=item["subject"],
            description=item["description"],
        ).with_inputs("subject", "description")
        if i < 12:
            trainset.append(ex)
        else:
            testset.append(ex)
    return trainset, testset

Optimization uses BootstrapFewShotWithRandomSearch with conservative settings: 2 bootstrapped demos, 2 labeled demos, 3 candidate programs, single-threaded (rate-limited embedding calls cannot be parallelized on the free tier).

Key observations

  • Server rack scores highest after optimization (0.672) — server racks are naturally rich in box-drawing structure (multiple horizontal rack units, vertical rails). This validates that the metric rewards structural complexity aligned with the computer corpus.
  • Joystick scores lowest (0.474 baseline) — a joystick is a simple shape (stick + base) with less structural density. The metric correctly reflects lower visual detail and semantic overlap with the computer corpus.
  • Bootstrap optimization does not improve average (57.0% vs 56.8%) — the delta is within noise. The baseline model (gpt-5.4-mini) already generates recognizable hardware art with its default instruction-following capability. Bootstrap's few-shot demonstrations do not add meaningful signal.
  • Speaker regresses under optimization (0.680 to 0.542) — the bootstrapped demos may steer the model toward a style that scores well on training subjects but poorly generalizes to speaker geometry.
  • The metric validates interleaved embedding discrimination — all 6 test subjects score well above 0.0 (the noise floor), confirming that ASCII art of hardware subjects lands near the computer corpus centroid, not in the noise region.

Insights

Insight

The interleaved embedding correctly discriminates structural complexity. A server rack drawn with box-drawing characters — multiple horizontal lines, vertical bars, nested brackets — scores higher than a joystick — a single stick on a base. This ordering matches human intuition about which hardware subjects produce richer ASCII art.

Insight

The triangular evaluation framework (noise — blank — computer) provides absolute positioning in embedding space, not just relative similarity. A score of 0.60 means the art is 60% of the way from the noise floor to the computer centroid. This is interpretable across subjects: a 0.60 joystick and a 0.60 server rack are equally "computer-like" in embedding space, even though they look completely different.

Insight

The test subjects are held-out hardware that the reference corpus has never seen. That all 6 score above 0.47 demonstrates that the computer centroid generalizes — the embedding captures "computer hardware-ness" as a semantic concept, not just visual similarity to the 6 specific reference pieces.

Limitations

Warning

Optimization flat. BootstrapFewShotWithRandomSearch did not improve over baseline. The model already generates decent hardware art from its instruction-following capability alone. The optimizer has limited room to improve when the baseline is already competent. This is a ceiling effect, not a metric failure.

Warning

Rate limiting constrains scale. Gemini Embedding 2's free tier allows approximately 1000 calls per day per API key. Each evaluation of a single subject requires one embedding call. A full optimization run (3 candidate programs × 12 training examples + 6 test evaluations × 2) consumes ~90 calls. This limits the number of candidate programs and bootstrap iterations that are practical.

Warning

Reference corpus size. The computer centroid is computed from only 6 reference pieces. A larger corpus (20–30 pieces spanning more hardware categories) would produce a more stable centroid and potentially improve discrimination on edge cases.

Open Questions

  • MIPROv2 instruction optimization — Would optimizing the generation instructions (rather than few-shot demos) produce better hardware art? MIPROv2 rewrites the prompt itself, which could discover more effective character-choice strategies that Bootstrap cannot.
  • Negative control — How do non-computer subjects (animals, nature scenes, abstract patterns) score against the computer centroid? A well-calibrated metric should assign them scores near 0.0. This has not been tested.
  • Centroid stability — Would a larger reference corpus (20+ pieces) meaningfully change the centroid position, or has it converged with 6 pieces? Internal cohesion measurements suggest the corpus is reasonably tight, but convergence has not been formally tested.
  • Cross-evaluation transfer — Do optimization gains on computer-01 transfer to other ascii-bench evaluations? If bootstrap demos improve server-rack generation, do they also improve noise-01 scores (they shouldn't) or story-01 scores (they might)?
